# Сверка — План и архитектура приложения

> Рабочее название: **Сверка** (набор бухгалтерских утилит)
> Версия документа: 0.1 (черновик архитектуры)
> Дата: июнь 2026

---

## 1. Идея приложения

Web-приложение — это **единая оболочка (shell)** с набором независимых **функций-модулей**.
Пользователь:

1. Заходит в интерфейс.
2. В левой панели выбирает нужную функцию (например, «Сравнить таблицы»).
3. В правой верхней панели загружает данные (например, две таблицы Excel).
4. В средней панели нажимает кнопки промежуточных операций (например, «Проверить дубли», «Сравнить»).
5. В нижней панели видит результат и может **скачать** его в виде файла (Excel/CSV).

Ключевое требование: **добавление новой функции не должно требовать переписывания всего кода**.
Каждая функция — это самостоятельный модуль, который «регистрируется» в оболочке.

## 2. Анализ существующих приложений

В корне проекта уже есть два готовых приложения. Они станут первыми двумя функциями «Сверки».

### 2.1. SearchDiff — «Сравнение Excel таблиц»

Файлы:
```
SearchDiff/
├── index.html          # интерфейс + CSS (встроенный)
├── app.js              # вся логика
├── xlsx.full.min.js    # библиотека парсинга Excel (локальная копия)
└── README.md
```

Принцип работы:
- Загрузка двух файлов `.xlsx / .xls / .csv`.
- Очистка данных (`cleanData`) — удаление строк «итого/всего/total» и пустых.
- Агрегация по компании (`aggregateData`) — суммирование значений.
- Сравнение агрегатов (`compareAggregatedData`) — поиск расхождений.
- Вывод списка расхождений на экран.
- **Скачивания результата НЕТ** (это будем добавлять).

Технологии: HTML5 + CSS3 + Vanilla JS (ES6) + XLSX.js.

### 2.2. SearchFall — «Авансовый отчёт vs Банк»

Файлы:
```
SearchFall/
└── excel-compare.html   # ВСЁ в одном файле (HTML + CSS + JS, XLSX через CDN)
```

Принцип работы (двухшаговый workflow):
- **Шаг 1.** Загрузка «Авансового отчёта» → проверка дублей по паре «Дата + Сумма».
- **Шаг 2.** Загрузка «Банка» → сравнение по столбцу «Сумма», поиск строк без соответствия.
- Промежуточные кнопки: `Проверить дубли`, `Сравнить с Банком`.
- Вывод таблиц-результатов с подсветкой.
- **Скачивание результата в Excel** (`XLSX.writeFile`, `aoa_to_sheet`).
- Нормализация сумм (`normalizeSum`) и дат (`normalizeDate`), автоопределение шапки (`detectHeaderRow`).

Технологии: HTML5 + CSS3 + Vanilla JS (IIFE) + XLSX.js (CDN).

### 2.3. Общие выводы (что переиспользуем)

| Принцип | Откуда взять |
|---|---|
| Загрузка / чтение Excel/CSV (`FileReader` + `XLSX.read`) | оба |
| Очистка и нормализация данных | SearchFall (`normalizeSum/Date`), SearchDiff (`cleanData`) |
| Автоопределение шапки таблицы | SearchFall (`detectHeaderRow`) |
| Многошаговый workflow с промежуточными кнопками | SearchFall |
| Рендер результата в виде таблицы/списка | оба |
| Экспорт результата в файл (`XLSX.writeFile`) | SearchFall |
| Полностью клиентская работа (без сервера) | оба |

Вывод: **«Сверка» — статическое клиентское приложение (browser-only)**, без бэкенда.
Это идеально совместимо с генератором статических сайтов **Jekyll**.

## 3. Почему Jekyll и как организовать модульность

Jekyll — генератор статических сайтов: собирает HTML-страницы из шаблонов (layouts),
включаемых частей (includes), данных (`_data/*.yml`) и коллекций (`_collections`).

Идея для «Сверки»:
- **Оболочка (4 панели)** — это общий `layout`.
- **Каждая функция** — это отдельный документ коллекции `_functions` + один JS-модуль.
- **Реестр функций** — файл данных `_data/functions.yml` (или авто-сборка из коллекции).
  Чтобы добавить функцию — добавляешь её файлы и одну запись в реестр. Остальной код не трогается.

> Важно: Jekyll даёт удобную **сборку и шаблонизацию**, но сама логика функций работает
> в браузере на JavaScript. Если установка Ruby/Jekyll нежелательна, ту же модульность
> можно реализовать на чистом статическом HTML+JS (вариант «без сборки», см. §6).

## 4. Архитектура: 4 панели интерфейса

```
┌───────────────────────────────────────────────────────────────────┐
│                          ВЕРХНЯЯ ШАПКА (заголовок Сверка)          │
├──────────────┬────────────────────────────────────────────────────┤
│              │  ПАНЕЛЬ 1 — ВВОД ДАННЫХ                              │
│   ЛЕВАЯ      │  (загрузка файлов / параметры выбранной функции)    │
│   ПАНЕЛЬ     ├────────────────────────────────────────────────────┤
│   ВЫБОРА     │  ПАНЕЛЬ 2 — ПРОМЕЖУТОЧНЫЕ ОПЕРАЦИИ                  │
│   ФУНКЦИЙ    │  (кнопки шагов: «Проверить», «Сравнить» и т.д.)     │
│   (меню)     ├────────────────────────────────────────────────────┤
│              │  ПАНЕЛЬ 3 — РЕЗУЛЬТАТ                                │
│              │  (таблица/список + кнопка «Скачать результат»)      │
└──────────────┴────────────────────────────────────────────────────┘
```

| Панель | Назначение | DOM-контейнер |
|---|---|---|
| Левая | Список функций, переключение активной функции | `#sidebar` |
| Верх-право (1) | Ввод данных: загрузка таблиц, поля параметров | `#panel-input` |
| Средняя (2) | Кнопки промежуточных операций / шагов | `#panel-actions` |
| Нижняя (3) | Вывод результата + скачивание | `#panel-output` |

Оболочка ничего не знает о конкретной функции — она лишь предоставляет 4 контейнера
и API, через который функция в них «рисует» свой UI.

## 5. Контракт модуля-функции (ключ к расширяемости)

Каждая функция реализует единый интерфейс. Оболочка вызывает только эти методы —
поэтому добавление новой функции не затрагивает существующий код.

```js
// /assets/functions/<имя>.js
window.AccProg.register({
  id: 'compare-tables',          // уникальный идентификатор
  title: 'Сравнение таблиц',     // подпись в меню
  description: 'Сравнивает суммы по компаниям из двух таблиц',
  icon: '📊',

  // 1) Что рисуем в панели ввода (загрузка файлов / параметры)
  renderInput(ctx) { /* ctx.inputEl, ctx.state */ },

  // 2) Какие кнопки промежуточных операций показать
  actions: [
    { id: 'compare', label: 'Сравнить', run: (ctx) => { /* ... */ } },
    // { id: 'check-dups', label: 'Проверить дубли', run: ... },
  ],

  // 3) Как нарисовать результат (вызывается из run)
  renderOutput(ctx, result) { /* ctx.outputEl */ },

  // 4) Как экспортировать результат в файл
  exportResult(ctx, result) { /* XLSX.writeFile(...) */ },
});
```

`ctx` (контекст), который оболочка передаёт функции:
```js
ctx = {
  inputEl,    // DOM средней панели ввода
  actionsEl,  // DOM панели кнопок
  outputEl,   // DOM панели результата
  state,      // объект для хранения данных функции (загруженные таблицы и т.п.)
  utils,      // общие утилиты (см. §7)
}
```

Жизненный цикл:
```
выбор функции в меню
   -> shell.activate(id)
   -> очистка 3 панелей
   -> fn.renderInput(ctx)         // панель 1
   -> fn.actions -> кнопки        // панель 2
   -> [пользователь жмёт кнопку]
   -> action.run(ctx)             // логика + fn.renderOutput -> панель 3
   -> [кнопка «Скачать»]
   -> fn.exportResult(ctx, result)
```

## 6. Структура файлов проекта

### Вариант A — полноценный Jekyll (рекомендуется для роста проекта)

```
Сверка/
├── _config.yml                 # конфиг Jekyll
├── index.html                  # точка входа (layout: shell)
├── _layouts/
│   └── shell.html              # оболочка: 4 панели + подключение ядра
├── _includes/
│   ├── sidebar.html            # левое меню (генерится из _data/functions.yml)
│   ├── panel-input.html
│   ├── panel-actions.html
│   └── panel-output.html
├── _data/
│   └── functions.yml           # РЕЕСТР функций (мета: id, title, script, icon)
├── _functions/                 # коллекция функций (по 1 .md-описанию на функцию)
│   ├── compare-tables.md
│   └── advance-vs-bank.md
├── assets/
│   ├── css/
│   │   └── app.css             # стили оболочки + общие
│   ├── js/
│   │   ├── core.js             # ядро: AccProg.register, роутинг, рендер панелей
│   │   └── utils.js            # общие утилиты (Excel/CSV/нормализация/экспорт)
│   ├── functions/              # ЛОГИКА функций (по 1 .js на функцию)
│   │   ├── compare-tables.js   # перенос логики из SearchDiff
│   │   └── advance-vs-bank.js  # перенос логики из SearchFall
│   └── vendor/
│       └── xlsx.full.min.js    # библиотека Excel (локально)
├── PLAN.md                     # этот документ
└── README.md
```

### Вариант B — без сборки (чистый статический HTML+JS)

Та же логика, но без Ruby/Jekyll. `index.html` подключает `core.js`, который
загружает реестр `functions.json` и подтягивает скрипты функций.

```
Сверка/
├── index.html
├── functions.json             # реестр функций
├── assets/css/app.css
├── assets/js/core.js
├── assets/js/utils.js
├── assets/functions/compare-tables.js
├── assets/functions/advance-vs-bank.js
└── assets/vendor/xlsx.full.min.js
```

> Оба варианта используют **один и тот же контракт модуля (§5)**, поэтому переход
> между ними не требует переписывания функций.

## 7. Общие утилиты (`utils.js`) — переиспользуемое ядро

Собираем из обоих приложений лучшие части:

| Утилита | Назначение | Источник |
|---|---|---|
| `readFile(file)` | `FileReader` + `XLSX.read` → массив строк | оба |
| `sheetToObjects(sheet)` | чтение с заголовками (как SearchDiff) | SearchDiff |
| `sheetToMatrix(sheet)` | чтение «как есть», `header:1` (как SearchFall) | SearchFall |
| `detectHeaderRow(rows)` | автоопределение строки шапки | SearchFall |
| `normalizeSum(v)` | нормализация чисел (пробелы, запятая, мусор) | SearchFall |
| `normalizeDate(v)` | нормализация дат (serial / DD.MM.YYYY / ISO) | SearchFall |
| `cleanData(rows)` | отброс «итого/всего/заголовок» | SearchDiff |
| `exportXlsx(rows, name)` | сохранение в `.xlsx` | SearchFall |
| `exportCsv(rows, name)` | сохранение в `.csv` | новое |
| `renderTable(el, headers, rows)` | универсальный рендер таблицы | оба |
| `showError / clearError` | сообщения об ошибках | оба |

## 8. План реализации (поэтапно)

1. **Каркас оболочки** — `index.html` + `app.css` + 4 панели + `core.js` (реестр, роутинг).
2. **Утилиты** — `utils.js` (чтение/нормализация/экспорт/рендер таблиц).
3. **Функция №1 «Сравнение таблиц»** — портирование логики SearchDiff в `compare-tables.js`
   по контракту §5, + добавить экспорт результата.
4. **Функция №2 «Авансовый отчёт vs Банк»** — портирование SearchFall в `advance-vs-bank.js`
   (двухшаговый workflow, дубли + сравнение, экспорт уже есть).
5. **Реестр функций** — `functions.yml` / `functions.json`; меню строится автоматически.
6. **(Опционально) Jekyll-обвязка** — layouts/includes/_config.yml для удобной сборки.
7. **Документация** — обновить README, добавить инструкцию «как добавить новую функцию».

### Как добавить новую функцию (целевой сценарий)
1. Создать `assets/functions/<id>.js` по контракту §5.
2. Добавить запись в реестр (`functions.yml` / `functions.json`).
3. (Jekyll) при желании добавить `_functions/<id>.md` с описанием.
4. Готово — функция появляется в меню, остальной код не меняется.

## 9. Технологический стек

- **HTML5 / CSS3** — оболочка и панели (CSS Grid для 4-панельной раскладки).
- **Vanilla JavaScript (ES6+)** — ядро и модули функций (без фреймворков).
- **XLSX.js (SheetJS)** — чтение и запись Excel/CSV (локальная копия в `vendor/`).
- **Jekyll** (опционально) — статическая сборка и шаблонизация.
- Без бэкенда: всё работает в браузере, данные не покидают компьютер пользователя.
