# CLm04 — Финальный проект

---

## Цель финального проекта

Финальный урок курса по Claude Code посвящён созданию проекта для портфолио.

> Наша задача — пройти путь от идеи до готового, работающего результата, который можно использовать в реальной работе.

**Финальный проект** — это демонстрация знаний и навыков через решение реальной задачи. Он показывает, что вы не просто смотрели уроки, а научились создавать полноценные рабочие инструменты.

> Финальный проект — это не отписка ради сертификата, а демонстрация ваших навыков через реальную задачу.

Проект должен быть полезен лично вам: автоматизировать рутину, ускорить рабочие процессы, освободить время для более важных задач. Лучший ориентир — **рабочие задачи**, потому что именно в них скрыт наибольший потенциал для автоматизации с помощью вайб-кодинга.

## Что даёт финальный проект

Завершённый проект решает сразу несколько задач:

- становится **первым кейсом в портфолио** (или очередным — если вы уже создавали проекты в ходе курса)
- может служить **основой для фриланс-услуг** или внутренней автоматизации
- демонстрирует заказчикам и клиентам ваш реальный уровень
- помогает ускорить и улучшить рабочие процессы, возможно — сократить рабочий день

> Все проекты, созданные в ходе курса, можно собрать в единую библиотеку и хранить в одном месте — чтобы в любой момент поделиться ими или использовать повторно.

## Формат проекта — MVP

Мы создаём не масштабный продукт, а **MVP** — минимально жизнеспособный продукт. Это версия проекта, которая уже функционирует и решает вашу задачу, но может иметь нюансы для доработки в будущем.

На создание финального проекта отводится **две недели**: этого достаточно, чтобы реализовать, продемонстрировать проект и получить сертификат завершения обучения.

## Как выбрать идею для проекта

### Шаг 1 — Найти рутинную задачу

Используйте проверенный подход: возьмите свои повседневные задачи и разбейте их на части.

- задачи, которые вы выполняете каждый день или часто
- задачи, которые отнимают много времени или кажутся скучными
- задачи, которые требуют точности и выглядят повторяемыми

Именно рутинные процессы — лучшие кандидаты для автоматизации с помощью вайб-кодинга.

### Шаг 2 — Сформулировать идею и ограничить её до MVP

После того как задача выбрана, сформулируйте: что именно автоматизируете? Что будет на входе? Что получится на выходе? Какой первый функционал?

> Важно: не пытайтесь охватить всё сразу. Один сценарий — уже полноценный MVP.

### Шаг 3 — Проверить идею по критериям

Актуальный финальный проект должен соответствовать трём критериям:

- **Понятная** — задача ясна и только проблему процесса: есть конкретный входной и выход
- **Завершённая** — есть конкретный результат, который можно увидеть и показать
- **Достижимая за две недели** — с учётом всех ресурсов и навыков

## Идеи выпускного проекта для курса по Claude Code

Ниже — примеры проектов, которые можно реализовать за две недели:

### Продажи
- **Лендинг-визитка для эксперта** — Сайт: быстро: portfolio для эксперта, на лендинге: резюме, описание услуг, блог, контактная форма, CTA для записи
- **Помощник для обработки заявок** — Скрипт или бот: обрабатывает входящие заявки: сортирует, классифицирует, отправляет ответ или уведомление
- **Автоматизация рассылок кронштейна** — Бот: автоматически распределяет задачи по базе данных

### Контент
- **Помощник для создания контента** — Сервис: помогает генерировать посты, описания, заголовки для блогов или соцсетей
- **Конструктор структуры урока** — Сервис: помогает собрать структуру урока по заданной теме

### Образование
- **Помощник для домашних заданий** — Сервис: помогает проверять и объяснять домашние задания
- **Проверка текстов** — Сервис: проверяет тексты на ошибки, стилистику, соответствие требованиям

### Аналитика
- **Анализатор файлов** — Сервис: анализирует CSV-файлы или другие табличные данные
- **Дашборд личных или рабочих показателей** — Сервис: собирает данные и строит графики

### Сервисы
- **Генератор ответов клиентам** — Сервис: помогает быстро готовить ответы на типовые обращения
- **Клиентский сервис** — Сервис: помогает обрабатывать обращения клиентов

### Персональная эффективность
- **Помощник для расписания задач** — Сервис: помогает планировать день и распределять задачи

### Прочее
- **Трекер задач** — Сервис: помогает вести список задач и отслеживать их выполнение
- **Поиск по базе знаний** — Сервис: помогает быстро находить информацию в базе знаний

## Как оформить описание проекта

### Структура описания проекта

Проект нужно не только создать, но и правильно оформить. Описание проекта строится по следующей структуре:

- **Название** — короткое, ёмкое, сразу понятное. Например: «Автогенератор презентаций по шаблону»
- **Проблема** — какую задачу решает проект и почему она важна
- **Пользователь** — для кого создаётся проект (методисты, маркетологи, команда, конкретная аудитория)
- **Основная функция** — что именно делает сервис, по пунктам: что подаётся на вход, что получается на выходе, какие инструменты задействованы
- **Формат результата** — что получит пользователь: готовая презентация, отчёт, статистика
- **Минимальный функционал** — что будет реализовано в первой версии (MVP); здесь же можно отметить, что планируется добавить в будущем
- **Технологии** — какой язык программирования использован, какие библиотеки, форматы ответов

> Описание можно составить **до начала разработки** — это поможет Claude Code лучше понять задачу и сразу создать проект под вашу аудиторию, проблему и ограничения. Если не знаете, как описать технологии — просто спросите Claude Code: он подробно расскажет, что было использовано в проекте.

### Описание как инструмент разработки

Подробное описание с болями пользователя, ожидаемыми результатами и ограничениями — это не просто документация. Это **контекст для Claude Code**, который позволяет ему с первого промпта адаптировать проект под конкретные задачи. Чем подробнее описание, тем точнее и качественнее будет результат разработки.

**Структура описания:**

- **Название** — короткое и понятное
- **Проблема** — какую задачу решает проект
- **Пользователь** — для кого создаётся
- **Основная функция** — что сервис должен делать
- **Формат результата** — что видит пользователь на выходе
- **Минимальный функционал** — что должно быть в первой версии
- **Технологии** — Python, Flask, HTML/CSS и т.д.

> **Ключевая мысль:** Claude Code работает лучше, когда получает не «сделай проект», а описание задачи, пользователя, результата и ограничений.

## Как разрабатывать проект

### Этапы разработки

Процесс создания финального проекта выглядит так:

1. Создаём **рабочую папку** — называем её на английском языке (русский текст в названии может вызвать ошибки в Claude Code)
2. Просим Claude Code **составить план** и разработать структуру проекта
3. **Набрасываем код** и создаём MVP
4. **Запускаем локально** и проверяем работу
5. **Дорабатываем**: улучшаем интерфейс, тексты, добавляем функционал
6. Готовим **README-инструкцию**
7. При желании — **публикуем проект** на виртуальном сервере

> Ориентировочный тайминг: первые 2–3 дня уходят на создание MVP, следующие полторы недели — на доработку и стабилизацию, ещё пара дней — на оформление.

### Локально vs. публикация

Проект может работать **локально** — этого достаточно для финального проекта. В этом случае демонстрация проходит через скриншоты каждого шага.

Публикация на **виртуальном сервере** не обязательна, но значительно усиливает впечатление: вместо скриншотов можно прислать ссылку, по которой проект можно открыть, покликать и взаимодействовать с ним в реальном времени.

> Для портфолио важно не только то, что проект существует, но и то, что его можно показать: как он выглядит, какой у него интерфейс, какие даёт результаты.

## Критерии хорошего финального проекта

### Что проверяется в итоговом проекте

В вайб-кодинге руки связаны — нет жёстких технических требований вроде «обязательно использовать две нейросети». Главное — чтобы проект решал реальную задачу. Тем не менее, есть ряд критериев, на которые стоит ориентироваться:

- **Понятная идея** — сразу ясно, что делает проект и зачем он нужен
- **Рабочий локальный запуск** — проект стабильно работает, каждый шаг можно продемонстрировать скриншотом
- **Минимальный интерфейс** — есть визуальная оболочка для взаимодействия
- **README с инструкцией запуска** — описание проекта и пользовательский сценарий
- **Демонстрируемость** — можно показать, как проект работает, без слов «ну, это предположительно работает»
- **Чистая структура файлов** — нет лишних файлов и «мусора» в коде
- **Ссылка на GitHub-репозиторий** — проект опубликован и доступен

> Слишком простой проект (например, базовый лендинг без дополнительного функционала) может не продемонстрировать ваши навыки в полной мере. Если делаете лендинг — усложните его: добавьте галереи, видео, автообновление или автогенератор контента.

### Скилл проекта

Помимо репозитория, стоит создать **скилл** для своего проекта — специальный файл, с помощью которого коллеги смогут развернуть точно такой же проект у себя. Это удобный способ делиться работой внутри команды.

> Сильный проект — не самый сложный, а тот, который можно открыть, понять, запустить и показать.

## GitHub — оформление профиля и публикация проекта

### Зачем нужен GitHub

> **GitHub** — это витрина проекта: здесь хранится код, описание, инструкции и доказательства выполненной работы. Это не только инструмент разработчиков — с его помощью можно демонстрировать проекты заказчикам и клиентам, делиться репозиториями с командой.

Мы создаём профиль на GitHub, публикуем туда финальный проект и оформляем его так, чтобы любой, кто зайдёт на страницу, сразу понял, кто вы, что умеете и что создали.

### Создание и оформление профиля

1. Переходим на **github.com** → нажимаем *Sign up for GitHub* → проходим регистрацию
2. Нажимаем на иконку профиля → **Profile**
3. **Edit profile**
4. Заполняем: имя (лучше на английском), должность или роль, компания, локация, email, ссылки на соцсети
5. Загружаем аватар (если фото слишком большое — сжимаем через онлайн-сервис сжатия изображений)
6. Сохраняем изменения

> Если вы в поиске работы или на фрилансе — так и напишите в описании профиля. Это помогает работодателям и заказчикам сразу понять, с кем они имеют дело.

### README-файл профиля

Чтобы профиль стал запоминающимся, добавляем README-файл — он отображается прямо на главной странице профиля.

**Как создать:**

1. Копируем свое имя пользователя (username) на GitHub
2. Создаём новый репозиторий с тем же именем, что и username
3. Переведя репозиторий в Public, добавляем README-файл → *Create repository*
4. Переходим в Claude и просим сгенерировать README в формате Markdown с описанием: чем занимаетесь, какие проекты делаете, ваш стек
5. Копируем готовый Markdown-текст → вставляем в README-файл репозитория → *Commit changes*

> Если есть файл **claude.md** с описанием себя — Claude сам подтянет информацию оттуда. Если нет — попросите его провести с вами интервью: задать вопросы и на основе ваших ответов составить текст.

> **Важное правило:** имя репозитория должно точно совпадать с вашим username на GitHub — иначе README не отобразится на главной странице профиля.

## Публикация проекта на GitHub через GitHub CLI

### Установка и авторизация GitHub CLI

Чтобы переносить локальные файлы в GitHub прямо с компьютера, нужен **GitHub CLI** (gh).

1. Переходим на страницу **GitHub CLI** → скачиваем версию для своей операционной системы → устанавливаем (Next → Next → Install)
2. Открываем **терминал** (командную строку)
3. Вводим команду для авторизации: `gh auth login`
4. Выбираем **GitHub.com** → **HTTPS** → подтверждаем (`Y`)
5. Следуем инструкциям: вводим код из терминала в браузере для подтверждения
6. После успешной авторизации Claude Code получает доступ к GitHub

### Публикация проекта

1. Открываем папку с проектом в Claude Code
2. Просим Claude Code опубликовать проект из этой папки в аккаунт GitHub как новый репозиторий и написать README.md с подробным описанием
3. В промпт можно добавить всю структуру описания проекта (название, проблема, пользователь и т.д.)
4. Claude Code создаёт README-файл и публикует репозиторий
5. Переходим на GitHub → раздел **Repositories** → видим опубликованный проект

### Добавление скриншотов в README

1. На странице репозитория нажимаем *Add file* → *Create new file*
2. В имени файла пишем `images/` — это создаёт папку для скриншотов
3. Также пишем название файла (любое). *Commit changes*
4. В папке `images` нажимаем *Add file* → *Upload files* → загружаем скриншоты проекта
5. *Commit changes* — сохраняем
6. Переходим в README-файл → нажимаем карандаш (редактировать)
7. Добавляем изображение в Markdown-синтаксисе: `![описание картинки](images/имя_файла.png)`
8. *Commit changes*

> Скриншоты — ключевой элемент репозитория. Если проект не опубликован на сервере, именно по скриншотам куратор и заказчики смогут оценить вашу работу.

## Создание и публикация скилла проекта

### Что такое скилл и зачем он нужен

> **Скилл** — это файл-инструкция для Claude Code, с помощью которого любой человек может развернуть у себя точно такой же проект. Это удобнее, чем просто давать ссылку на репозиторий: коллеги получают готовую точку входа и могут сразу начать работать с проектом.

Скилл описывает: структуру репозитория, как запустить сервис, как вносить изменения, как проверять ошибки, как готовить проект к публикации.

### Как создать и опубликовать скилл

1. Открываем командную строку. Переходим в Claude Code. Переходим в папку с проектом в Claude Code
2. Используем промпт: «Создай скилл для моего проекта. Он должен помогать Claude Code работать с этим репозиторием: понимать структуру, запускать сервис, вносить изменения, обновлять README, проверять ошибки и готовить проект к публикации»
3. Claude Code создаёт папку `.claude/` со скиллом внутри
4. Просим Claude Code опубликовать папку со скиллом в тот же репозиторий на GitHub
5. Проверяем: в репозитории появляется раздел с готовым скиллом

После публикации финальный репозиторий содержит всё необходимое:

- подробное **README** с описанием проекта
- **скриншоты** работающего проекта
- **скилл** для передачи коллегам
- **ссылку** на опубликованный проект (если есть)

> Ссылку на репозиторий можно открыть в режиме инкогнито и проверить, что всё отображается корректно — именно так это будет выглядеть для куратора или заказчика.

## Онлайн-выпускной и защита проекта

### Формат онлайн-выпускного

После двух недель работы над проектом проходит **онлайн-выпускной**, где каждый желающий может представить свою работу.

Выступление занимает **3–5 минут**.

**Структура презентации проекта:**

- **Название и короткое описание** — что это за проект
- **Проблема** — какую задачу вы решили
- **Живая демонстрация** — показываем, как проект работает (даже если он локальный)
- **Результаты** — что получается на выходе
- **GitHub** — показываем репозиторий
- **Планы** — как хотите развивать проект дальше или какие следующие проекты планируете реализовать с помощью вайб-кодинга

> Если ваш проект работает долго (например, генерация презентации занимает 10 минут) — покажите уже готовый результат и объясните, как он был получен. Заранее подготовьте финальный вывод проекта для демонстрации.

## Итоги

### Дополнительные материалы

#### Ссылки на сервисы из урока

- **GitHub** (github.com) — платформа для хранения и публикации репозиториев с кодом, документацией и скриншотами проектов
- **GitHub CLI** (cli.github.com) — утилита командной строки для работы с GitHub прямо с локального компьютера
- **Claude / Claude Code** (claude.ai) — AI-ассистент, используемый для разработки проектов, создания README, скиллов и оформления профиля
- **Онлайн-сервис сжатия изображений** — любой удобный сервис (например, squoosh.app или tinypng.com) для уменьшения размера фотографий перед загрузкой на GitHub

#### Словарь занятия

- **MVP (Minimum Viable Product)** — минимально жизнеспособный продукт; версия проекта с ограниченным, но работающим функционалом, которую можно оценить и использовать
- **Вайб-кодинг** — подход к разработке с помощью AI-инструментов (в первую очередь Claude Code), при котором код пишется и дорабатывается через диалог с ассистентом
- **Репозиторий** — хранилище файлов проекта на GitHub (код, README, скриншоты, скилл)
- **README.md** — текстовый файл в формате Markdown с описанием проекта: название, задача, инструкция по запуску, скриншоты
